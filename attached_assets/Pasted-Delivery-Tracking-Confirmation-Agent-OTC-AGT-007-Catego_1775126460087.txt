Delivery Tracking & Confirmation Agent
OTC-AGT-007  |  Category: Logistics
8.1 Description
Provides unified, real-time visibility into shipment status across multiple carriers and modes of transport. Proactively detects delivery risks, manages proof-of-delivery capture, triggers downstream processes upon confirmation, and enables customer self-service tracking.
8.2 Workflow Steps
66.	1. Receive shipment creation event from Fulfillment Agent
67.	2. Register tracking across all carriers (parcel, LTL, FTL, ocean, air)
68.	3. Poll or receive webhook updates on shipment milestones
69.	4. Normalize tracking events into unified status model
70.	5. Detect anomalies: delayed scan, route deviation, customs hold, temperature excursion
71.	6. Send proactive notifications at key milestones (shipped, in-transit, out-for-delivery, delivered)
72.	7. Capture proof-of-delivery (signature, photo, geolocation)
73.	8. Confirm delivery and update order status
74.	9. Trigger warranty activation, service setup, or survey if applicable
75.	10. Handle failed deliveries: reschedule, redirect, or return
76.	11. Archive tracking data for analytics and dispute resolution

8.3 Common Ontology
The following entities and attributes form the shared data model referenced by this agent:
Entity	Key Attributes
Tracking Event	timestamp, status code, location, carrier, description
Shipment Milestone	shipped, in-transit, customs, out-for-delivery, delivered, exception
Proof of Delivery	type (signature, photo, GPS), timestamp, recipient name
Carrier Integration	API type (REST, EDI, webhook), polling frequency, SLA
Delivery Exception	type (delay, damage, refused, lost), action required
Notification	channel (email, SMS, push), template, trigger condition

The following skills should be developed and made available as context for this agent:

8.5 Eval Dataset
Recommended evaluation data and benchmarks for validating agent accuracy and performance:
•	Historical tracking data with actual delivery times for ETA prediction validation
•	Anomaly cases: late deliveries that could have been detected earlier
•	Multi-carrier shipments: tracking accuracy across different carrier APIs
•	Notification timing: customer preference adherence and engagement rates
•	POD accuracy: correct delivery confirmation rate
•	Failed delivery resolution: success rate and cycle time

8.6 Production Runbooks
Operational procedures for handling common issues and failure scenarios:
Scenario	Procedure
Carrier API Down	Fallback polling schedule, manual tracking update procedure
Mass Delivery Delay (weather event)	Bulk ETA update, proactive customer notification
POD Dispute	Investigation procedure, carrier claim initiation
Tracking Data Gap	Carrier contact procedure, estimated status interpolation
International Customs Hold	Documentation requirements, broker escalation
Carrier Data Format Change	Integration update procedure, data mapping review
