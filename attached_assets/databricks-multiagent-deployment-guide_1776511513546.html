⚒Architecture Summary

There is exactly ONE Databricks Model Serving endpoint — the orchestrator. Sub-agents are Python wheels installed into the same environment and called in-process. No HTTP between agents, no separate endpoints.

User Request │ ▼ wealth-mgmt-orchestrator ← ONLY serving endpoint │ │ Python in-process calls (no HTTP, no separate endpoints) │ ├─── CustomerOnboardingAgent.predict() │ Stage 1: KYC/AML, identity docs, account creation │ ├─── InvestorSuitabilityProfilingAgent.predict() │ Stage 2: Risk tolerance, suitability profile │ ├─── InvestmentRecommendationExecutionAgent.predict() │ Stage 3: Recommend products, consent, execute investment │ └─── PortfolioTrackingGoalProgressAgent.predict() Stage 4: Monitor ROI vs benchmark, compliance report
Sub-agents are packaged as .whl files and installed as DAB environment dependencies.
All agents use the ResponsesAgent pattern from mlflow.pyfunc (MLflow ≥ 2.20.0).
The spark_python_task only registers the orchestrator model; sub-agents ride along as imports.
Credentials (DATABRICKS_HOST, DATABRICKS_TOKEN) are auto-injected by Model Serving — never hardcoded.

01
catalog/schema not valid under workspace in databricks.yml
YAML / DAB
▶
Wrong
Adding catalog: workspace and schema: default inside the workspace: block of databricks.yml. These fields do not exist in the DAB spec and cause a validation error on databricks bundle deploy. 
workspace:
  catalog: workspace   # ❌ not a valid DAB field
  schema: default      # ❌ not a valid DAB field
  root_path: ~/.bundle/...
Fix
Remove those lines entirely. Set the Unity Catalog coordinates in your Python agent code using MLflow APIs. Use registered_model_name="catalog.schema.model_name" in log_model(). 
# In agent.py __main__ block:
import mlflow
mlflow.set_tracking_uri("databricks")
mlflow.set_registry_uri("databricks-uc")

mlflow.pyfunc.log_model(
    artifact_path="agent",
    registered_model_name="workspace.default.my_agent",  # ✅ UC coordinates here
    ...
)
02
Serverless compute only workspace — job_clusters config invalid
YAML / DAB Infra
▶
Wrong
Specifying a job_clusters block with node_type_id, num_workers: 0, or spark_version. Serverless-only workspaces reject classic cluster config entirely. 
job_clusters:                        # ❌ not valid on serverless workspaces
  - job_cluster_key: agents_cluster
    new_cluster:
      node_type_id: Standard_D4s_v3
      num_workers: 0
      spark_version: 14.3.x-scala2.12
Fix
Remove the job_clusters block entirely. Serverless compute is automatically used when no cluster is specified. Use environments with spec.client: "2" to declare Python dependencies. 
environments:                        # ✅ serverless environment spec
  - environment_key: agents_env
    spec:
      client: "2"
      dependencies:
        - mlflow>=2.20.0,<3.0.0
        - ./dist/my_agent-0.1.0-py3-none-any.whl

tasks:
  - task_key: deploy_agent
    spark_python_task:
      python_file: ${workspace.file_path}/agent.py
    environment_key: agents_env
    # NO job_cluster_key — serverless is the default

requires-python = ">=3.12" incompatible with serverless Python 3.11
Dependencies Infra
▶
Wrong
All pyproject.toml files had requires-python = ">=3.12". Databricks serverless runs Python 3.11. Pip refuses to install any wheel that declares a 3.12+ requirement, causing the task to fail at dependency resolution. 
[project]
requires-python = ">=3.12"   # ❌ serverless is 3.11
Fix
Lower the constraint to >=3.11 in every agent's pyproject.toml. This applies to the orchestrator and all four sub-agent packages. 
[project]
requires-python = ">=3.11"   # ✅ matches Databricks serverless
05
Exact version pins cause conflicts with pre-installed packages
Dependencies
▶
Wrong
Using exact version pins like openai==1.30.0, mlflow==2.18.0, pytest==8.3.4 in main dependencies. Serverless has pre-installed packages at specific versions; exact pins conflict with the pre-installed set and cause pip backtracking or outright failure. 
dependencies = [
    "openai==1.30.0",     # ❌ exact pin
    "mlflow==2.18.0",     # ❌ exact pin
    "pytest==8.3.4",      # ❌ exact pin
]
Fix
Use loose lower-bound constraints. For packages that have a known breaking change, add an upper bound to stay safe. 
dependencies = [
    "openai>=1.30.0",          # ✅ lower-bound only
    "requests>=2.31.0",        # ✅ lower-bound only
]

[project.optional-dependencies]
dev = [
    "mlflow>=2.18.0",
    "pytest>=7.4.0,<9.0.0",    # ✅ bounded range in dev only
]
06
MLflow pre-installed version (2.11.4) is too old for ResponsesAgent
MLflow Dependencies
▶
Wrong
Assuming the serverless environment has a recent enough MLflow. Databricks serverless pre-installs MLflow 2.11.4. ResponsesAgent requires MLflow ≥ 2.20.0. If you do not explicitly upgrade it in the DAB environment spec, the job silently uses the old version and fails with ImportError or AttributeError.
Fix
Explicitly add mlflow>=2.20.0,<3.0.0 as the first dependency in your DAB environment spec so it is upgraded before the job code runs. What actually gets installed is 2.22.4 (latest in that range as of this writing). 
environments:
  - environment_key: agents_env
    spec:
      client: "2"
      dependencies:
        - mlflow>=2.20.0,<3.0.0   # ✅ must be first — upgrades from 2.11.4
        - ./dist/my_agent-0.1.0-py3-none-any.whl

11
Top-level imports execute during spark_python_task exec() before upgraded MLflow is active
Python Infra
▶
Wrong
Placing all MLflow / ResponsesAgent imports at the module top level. When the DAB job runs spark_python_task, it calls Python's built-in exec() on the file. Top-level imports execute immediately at that point — but the upgraded MLflow environment (from the environments spec) may not be fully active yet in the exec context. This causes ImportError on ResponsesAgent. 
# ❌ top-level — executes during spark_python_task exec()
import mlflow
from mlflow.pyfunc import ResponsesAgent
from mlflow.types.responses import ResponsesRequest, ResponsesStreamEvent
from openai import OpenAI
Fix
Guard all serve-time imports inside if __name__ != "__main__":. The spark_python_task only runs the __main__ block, which only needs plain mlflow (pre-installed) and databricks-sdk. The serve-time block runs later, inside the serving container where the upgraded environment is fully in effect. 
# ✅ serve-time block — runs only inside the Model Serving container
if __name__ != "__main__":
    import mlflow
    from mlflow.pyfunc import ResponsesAgent
    from mlflow.types.responses import ResponsesRequest, ResponsesResponse, ResponsesStreamEvent
    from openai import OpenAI
    # ... define agent class, set_model ...

# ✅ job entry point — runs in spark_python_task
if __name__ == "__main__":
    import mlflow
    mlflow.set_tracking_uri("databricks")
    mlflow.set_registry_uri("databricks-uc")
    # ... log_model, create endpoint ...

MLflow Infra
▶
Wrong
Calling mlflow.set_experiment() or mlflow.start_run() without first setting tracking and registry URIs. On serverless, the Spark configuration key spark.mlflow.modelRegistryUri is not populated, so MLflow defaults to local file tracking and fails when it tries to register to Unity Catalog.

17
WorkspaceClient() fails on Model Serving — no ambient credentials
Python Infra
▶
Wrong
Instantiating WorkspaceClient() inside the agent class (serve-time) to get an OpenAI client via .serving_endpoints.get_open_ai_client(). The Databricks SDK credential chain looks for a Databricks config profile or environment variables, but in the Model Serving container the credentials are in DATABRICKS_HOST and DATABRICKS_TOKEN — the SDK does not always discover them in time, and this call sometimes raises ValueError: credentials not found. 
def _get_llm_client(self):
    # ❌ WorkspaceClient() may fail to find credentials on Model Serving
    return WorkspaceClient().serving_endpoints.get_open_ai_client()
Fix
On Model Serving, DATABRICKS_HOST and DATABRICKS_TOKEN are always auto-injected as environment variables. Build the OpenAI client directly from them. 
def _get_llm_client(self) -> OpenAI:
    if self._llm_client is None:
        host  = os.environ.get("DATABRICKS_HOST", "").rstrip("/")
        token = os.environ.get("DATABRICKS_TOKEN", "")
        self._llm_client = OpenAI(
            base_url=f"{host}/serving-endpoints",   # ✅ direct from env vars
            api_key=token,
        )
    return self._llm_client

log_model calls predict() on input_example — fails without DATABRICKS_HOST
MLflow Python
▶
Wrong
Having predict_stream unconditionally attempt a real LLM call. When MLflow calls log_model() with an input_example, it calls predict() internally to validate the schema. This happens inside the spark_python_task context where DATABRICKS_HOST is not set (the serving endpoint doesn't exist yet), causing the OpenAI client to fail with a connection error.
Fix
Guard the real LLM call with an environment check. Return a stub response when DATABRICKS_HOST is absent (i.e., during model logging validation). 
def predict_stream(self, request: ResponsesRequest):
    # Guard: during mlflow.pyfunc.log_model, DATABRICKS_HOST is not set.
    # Return a stub so schema validation succeeds without a real LLM call.
    if not os.environ.get("DATABRICKS_HOST"):
        yield ResponsesStreamEvent(
            type="response.output_item.done",
            item={
                "type": "message",
                "id": str(uuid4()),
                "role": "assistant",
                "status": "completed",
                "content": [{"type": "output_text", "text": "stub"}],
            }
        )
        return
    # Real logic follows...
    yield from self._run_agentic_loop(...)

Architecture Infra
▶
Wrong
Planning to deploy five separate Databricks Model Serving endpoints — one for the orchestrator and one for each of the four sub-agents — and have the orchestrator call sub-agents over HTTP. 22
AutoCaptureConfigInput rejected for agent model endpoints
MLflow Infra
▶
Wrong
Passing auto_capture_config=AutoCaptureConfigInput(...) to create_and_wait() or update_config_and_wait() when the endpoint serves an agent (MLflow ResponsesAgent) model. Databricks rejects this with an InvalidParameterValue error because the inference-table contract for agent endpoints is different. 
from databricks.sdk.service.serving import (
    ServedEntityInput, EndpointCoreConfigInput, AutoCaptureConfigInput,
)
auto_capture_config = AutoCaptureConfigInput(
    catalog_name="workspace", schema_name="default", enabled=True,
)
# ❌ InvalidParameterValue: For endpoints serving agent models,
#    please use the 'ai_gateway' field to manage inference tables.
Fix
Use AiGatewayConfig + AiGatewayInferenceTableConfig and apply them via a separate put_ai_gateway() call after the endpoint is created or updated. The ai_gateway field does not exist on EndpointCoreConfigInput — it must be set through the dedicated API. 
from databricks.sdk.service.serving import (
    ServedEntityInput, EndpointCoreConfigInput,
    AiGatewayConfig, AiGatewayInferenceTableConfig,
)
_AI_GATEWAY = AiGatewayConfig(
    inference_table_config=AiGatewayInferenceTableConfig(
        catalog_name="workspace",
        schema_name="default",
        enabled=True,
    )
)
# Create or update endpoint first (no auto_capture_config argument)
w.serving_endpoints.create_and_wait(
    name=_SERVING_ENDPOINT_NAME,
    config=EndpointCoreConfigInput(
        name=_SERVING_ENDPOINT_NAME,
        served_entities=[served_entity],
    ),
    timeout=timedelta(minutes=30),
)
# Then apply ai_gateway separately
w.serving_endpoints.put_ai_gateway(
    name=_SERVING_ENDPOINT_NAME,
    inference_table_config=_AI_GATEWAY.inference_table_config,
)  # ✅ correct for agent model endpoints

Inference tables not supported for this endpoint type in workspace
Infra
▶
Wrong
Even after switching to the correct put_ai_gateway() API (Issue 22), calling it may raise a NotFound error in some Databricks workspace tiers: "Inference table is not currently supported for this endpoint type in this workspace." Inference tables are a premium feature gated by workspace tier and region. 
w.serving_endpoints.put_ai_gateway(
    name=_SERVING_ENDPOINT_NAME,
    inference_table_config=...,
)
# ❌ NotFound: Inference table is not currently supported
#    for this endpoint type in this workspace.
Fix
Drop the put_ai_gateway() call entirely when the workspace does not support inference tables. MLflow traces still work correctly through the MLFLOW_TRACKING_URI=databricks and MLFLOW_EXPERIMENT_NAME environment variables set on the endpoint — those route traces to the Databricks workspace tracking server independent of inference tables. 
from databricks.sdk.service.serving import ServedEntityInput, EndpointCoreConfigInput

_ENDPOINT_ENV_VARS = {
    "MLFLOW_TRACKING_URI":    "databricks",   # routes traces to workspace server
    "MLFLOW_EXPERIMENT_NAME": _EXPERIMENT_NAME,
}
served_entity = ServedEntityInput(
    entity_name=_REGISTERED_MODEL_NAME,
    entity_version=str(model_info.registered_model_version),
    workload_size="Small",
    scale_to_zero_enabled=True,
    environment_vars=_ENDPOINT_ENV_VARS,   # ✅ traces work without inference tables
)
# No put_ai_gateway() call needed — or can be wrapped in try/except
if _SERVING_ENDPOINT_NAME in existing:
    w.serving_endpoints.update_config_and_wait(

else:

        name=_SERVING_ENDPOINT_NAME,

            name=_SERVING_ENDPOINT_NAME,

⚙Part 2: Architecture Evolution & Code Optimizations

After resolving the initial 21 deployment blockers, the codebase underwent significant architectural refactoring. Each change below reduces complexity, cost, or latency — and prevents entire classes of future bugs. Every card documents the old approach, the new approach, and the reasoning.

A1
Single Endpoint Architecture — Sub-agents as In-Process Libraries
Architecture Infra
▶
Old Approach
5 separate Databricks Model Serving endpoints — one per agent. The orchestrator called sub-agents over HTTP, meaning every request required up to 4 additional network round-trips inside Databricks infrastructure.
New Approach
ONE endpoint — the orchestrator only. Sub-agents are Python libraries called in-process via the BaseWealthAgent mixin. No HTTP hops, no separate endpoint auth, no separate deployment pipelines. 
# Sub-agents are called as plain Python classes — zero network overhead
result = CustomerOnboardingAgent().predict(sub_request)
result = InvestorSuitabilityProfilingAgent().predict(sub_request)

Why: No exposure risk, lower cost (one serving endpoint billed), lower latency (no HTTP hops between agents), and no inter-agent authentication to configure.

A2
BaseWealthAgent — Shared Agentic Loop via Mixin
Python Architecture
▶
Old Approach
Each of the 5 agent files had 165+ lines containing a full duplicate of the agentic loop (LLM call, tool dispatch, streaming, tracing). Any bug fix or enhancement had to be applied to 5 separate files.
New Approach
shared/base_agent.py contains the loop once. Each agent declares only its identity — system prompt, tools, and endpoint name — in 4 lines: 
class CustomerOnboardingAgent(BaseWealthAgent, ResponsesAgent):
    def __init__(self):
        self.llm_endpoint = _LLM_ENDPOINT
        self.system_prompt = _SYSTEM_PROMPT
        self._tools_dict = {t.name: t for t in load_tools()}
        self._deploy_client = None

Why: DRY principle. Adding a new agent is 4 lines, not 165. Any loop change (tracing, error handling, streaming format) propagates to all agents automatically.

A3
agents.yaml Registry — Single Source of Truth for Agent Identity
YAML / DAB Architecture
▶
Old Approach
Each agent required its own agent.py + agent.yaml + pyproject.toml + Makefile = 14+ files just for identity. Adding a new agent meant creating and maintaining a full new directory of boilerplate.
New Approach
One agents.yaml at the project root. The create_agent("key") factory builds any agent from the registry. Adding a new agent is one YAML block: 
my_new_agent:
  name: "My New Agent"
  system_prompt: |
    You are...
  tools_module: "tools.agents.my_new_agent.tools"
  max_iterations: 5

Why: Zero Python files needed per agent. All agent configuration lives in one inspectable, diffable place. create_agent(key) is the only factory needed.

A4
Single Wheel Build — One pyproject.toml at Root
Dependencies Infra
▶
Old Approach
5 separate pyproject.toml files, 5 wheels built independently and uploaded to Databricks. Each deployment step required 5× build + 5× upload + 5× install.
New Approach
One pyproject.toml at the project root producing one wheel: wealth_management_bundle-0.1.0-py3-none-any.whl. All sub-agent tools are bundled inside via sync_bundle.py (see A5). 
# One command does everything
make deploy-dev
# → make sync → python scripts/sync_bundle.py
# → pip wheel . → dist/wealth_management_bundle-0.1.0-py3-none-any.whl
# → databricks bundle deploy

Why: One build, one upload, one install. Simpler CI/CD. No wheel version skew between agents.

A5
sync_bundle.py — Automated Pre-Build Sync
Python Infra
▶
Old Approach
Sub-agent tools had to be manually copied into the orchestrator wheel before every build. This was easy to forget, leading to stale tool code being deployed silently.
New Approach
scripts/sync_bundle.py auto-syncs all tools and fixes import paths before every build. The Makefile wires it in so it is impossible to forget: 
# Makefile

sync:
    python scripts/sync_bundle.py   # runs before every build automatically

Why: make deploy-dev calls make sync which calls python scripts/sync_bundle.py — impossible to deploy stale tool code.

A6
MLflow Tracing with @mlflow.start_span Context Managers
MLflow Python
▶
Old Approach
No tracing at all. LLM calls and tool executions were completely opaque — debugging required adding print statements and re-deploying.
New Approach
Every LLM call and tool execution is wrapped in a span. A parent AGENT span in predict() anchors all child spans in the call tree: 
# Parent span — anchors the full request trace
with mlflow.start_span(name="predict", span_type="AGENT") as span:
    span.set_inputs({"query": request.input[0].content})

    # Child span — each LLM call
    with mlflow.start_span(name="llm_call", span_type="LLM") as llm_span:
        llm_span.set_inputs({"num_messages": len(messages)})
        response = client.predict(endpoint=self.llm_endpoint, inputs={...})
        llm_span.set_outputs({"content_preview": content[:200]})

Why: Full call tree visible in Databricks Experiments → Traces tab. Latency per LLM call, per tool, per agent — all observable without code changes.

A7
get_deploy_client("databricks") — Universal Auth Without Env Vars
MLflow Infra
▶
Old Approach
Two failed attempts:
  1. WorkspaceClient() — failed on serving endpoint (no credentials in the SDK chain)
  2. OpenAI(base_url=os.environ['DATABRICKS_HOST']) — failed because DATABRICKS_HOST not set in all contexts
# WRONG — fails on serving endpoint (no default credentials)

# WRONG — fails in job clusters (DATABRICKS_HOST not always set)

New Approach
get_deploy_client("databricks") — MLflow's deployment client handles auth in ALL execution contexts automatically: 
from mlflow.deployments import get_deploy_client

    "messages": messages,
    "tools": tool_specs,
})

Why: Works everywhere without any environment variables — job cluster, serving endpoint, local dev. MLflow handles credential discovery internally.

A8
Intelligent Tool Routing — LLM Decides Which Agents to Call
Architecture Python
▶
Old Approach
The system prompt forced ALL 4 sub-agents to run every time in a fixed sequence, regardless of what the user actually requested. Every single query triggered all 4 agents — even "what is my portfolio value?" ran onboarding, suitability, and recommendation agents unnecessarily.
New Approach
The system prompt instructs the LLM to call ONLY the relevant sub-agent(s) based on the request context:
Routing examples
"I want to invest" → calls suitability + recommendation only
"Track my portfolio" → calls portfolio only
"New customer onboarding" → calls all 4 in order

Why: Up to 4x faster for simple requests. Correct behavior — users aren't subjected to irrelevant agent processing. Cost scales with request complexity, not always worst-case.

A9
MAX_ITERATIONS Reduced — 3x Fewer Worst-Case LLM Calls
Python Architecture
▶
Old Approach
All agents had MAX_ITERATIONS = 20. With 1 orchestrator + 4 sub-agents, the worst-case path was 100 LLM calls per request (20 × 5 agents).
New Approach
Orchestrator = 10 iterations, sub-agents = 5 iterations. Worst case is now 30 LLM calls (10 + 4×5), sufficient for all observed use cases. 
# In base_agent.py
_MAX_ITERATIONS_ORCHESTRATOR = 10
_MAX_ITERATIONS_SUB_AGENT    = 5   # overridden per-agent via agents.yaml max_iterations

Why: 3x faster worst-case latency and 3x lower worst-case token cost. All real use cases terminate well within these limits.

A10
predict() Returns Single Clean Response — No Intermediate Bubbles
Python MLflow
▶
Old Approach
predict() collected all stream events including intermediate tool call outputs: output = [e.item for e in predict_stream(...)]. This returned multiple items to the UI, causing multiple response bubbles to appear for a single request. 
# WRONG — returns all items including intermediate tool call outputs
def predict(self, request):
    output = [e.item for e in self.predict_stream(request) if hasattr(e, "item")]
    return ResponsesResponse(output=output)   # ← multiple bubbles in UI
New Approach
Extract only the final assistant text message. Return a single clean response: 
# RIGHT — returns only the final assistant message
def predict(self, request):
    final_text = ""
    for event in self.predict_stream(request):
        if hasattr(event, "item") and event.item:
            item = event.item
            if item.get("role") == "assistant":
                for c in item.get("content", []):
                    if c.get("type") == "output_text":
                        final_text = c.get("text", "")
    return ResponsesResponse(output=[{
        "type": "message", "id": str(uuid4()),
        "role": "assistant", "status": "completed",
        "content": [{"type": "output_text", "text": final_text}],
    }])

Why: The UI shows exactly one response bubble per request. Intermediate tool call outputs are tracing artifacts, not user-facing content.

A11
No File I/O at Module Load Time in agent.py
Python MLflow
▶
Old Approach
agent.py read agents.yaml at module load time using a relative path. When MLflow's log_model() validates the model, it runs agent.py in a temporary directory where relative paths don't resolve — causing FileNotFoundError.

Why: log_model runs agent.py in a temp directory where relative paths don't resolve. Lazy loading with __file__-anchored paths avoids the temp-dir trap entirely.

A12
_setup_mlflow_tracing — Idempotent Lazy Initialization
MLflow Python
▶
Old Approach

# WRONG — network call at module load fails during log_model validation

New Approach

class OrchestratorAgent(BaseWealthAgent, ResponsesAgent):

        _setup_mlflow_tracing("/Shared/wealth_mgmt")  # ✅ only on first real request
        ...

Why: Safe to call on every request, configures only once, never fails during log_model validation since the network call is deferred until the first real predict().

MLflow Version Compatibility Shim for ResponsesRequest Class Names
MLflow Python
▶
Old Approach

Why: MLflow 2.x uses ResponsesRequest, MLflow 3.x renamed them to ResponsesAgentRequest. The shim handles both transparently — codebase works on Databricks environments running either version.
