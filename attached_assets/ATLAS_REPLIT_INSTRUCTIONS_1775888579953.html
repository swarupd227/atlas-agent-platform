Atlas Agent Creator · Replit Implementation Guide

How to Build a Correct, Production-Ready Agent

Platform: Databricks Serverless + MLflow Audience: Replit Developer Applies to: Every agent created by Atlas Updated: April 2026

0 Purpose of This Document

Atlas generates agent scaffolding — folder structure, config files, and starter code. That scaffolding is a starting point, not a finished product. Before any agent is deployable, Replit must complete the implementation. This document defines exactly what "complete" means, regardless of what the agent does or what backends it connects to.

These instructions apply to every agent Atlas creates — whether it connects to MCP servers, REST APIs, knowledge bases, databases, third-party SaaS tools, or any combination. The patterns are the same; only the backend details differ.

1 The Two Things Atlas Gets Wrong (Always Fix These)

1a. The agent framework

Atlas may generate agents using older frameworks like LangChain PythonModel. The correct framework for all Databricks agents is MLflow ResponsesAgent. If the generated agent.py does not extend ResponsesAgent, replace it.

Wrong (replace this)Correct (use this)
Extends mlflow.pyfunc.PythonModel Extends mlflow.pyfunc.ResponsesAgent
Uses LangChain ChatDatabricks or bind_tools() Uses WorkspaceClient.serving_endpoints.get_open_ai_client()
Autolog: mlflow.langchain.autolog() Autolog: mlflow.openai.autolog()
Only implements predict() — blocking, no streaming Implements both predict() and predict_stream()
Dependencies: databricks-langchain, langchain-core Dependencies: openai>=1.30.0, requests>=2.31.0

1b. The tool stubs

Atlas generates stub implementations for every tool. They look like this:

Stub pattern — never acceptable in a deployed agent return {"status": "ok", "_stub": True, "message": "TODO: Replace this stub..."}

A stub returns fabricated data. The LLM will respond as if the tool worked, but nothing real happened. KYC was not checked. No record was created. No data was fetched. Stubs must be replaced with real implementations before the agent is useful.

2 Understanding What Each Agent Needs

Every agent is defined by an agent.yaml file. Before writing any code, read it carefully to understand:

  • What tools are declared — these are the capabilities the agent has
  • What external resources are listed — MCP servers, knowledge bases, APIs
  • What the system prompt says — the agent's domain, rules, and workflow
  • What KPIs are defined — the business outcomes the agent must achieve

Each tool in agent.yaml corresponds to a file in tools/. Each tool must connect to the resource(s) listed for it. The resource type determines how the implementation is written.

Possible resource types

🔌

MCP Server

JSON-RPC 2.0 over HTTP. Declared in mcp_servers in agent.yaml. Each server exposes named tools.

🌐

REST API

Standard HTTP endpoints. May be internal services, third-party APIs, or SaaS platforms with REST interfaces.

📚

Knowledge Base

Vector search index. Declared in knowledge_bases in agent.yaml. Used for semantic retrieval — policy docs, product catalogs, FAQs.

🗄️

Database / Data Warehouse

SQL queries against Databricks Unity Catalog, Delta tables, or external databases via JDBC.

⚙️

Unity Catalog UDF

Python functions registered in UC. Called via uc_function_client.execute_function(). Native Databricks governance and audit.

📊

Local / Computed Data

No external call needed. Tool computes or transforms data internally — e.g. risk scoring, formatting, calculations.

A single tool may use multiple resource types For example, a recommendation tool might query a knowledge base for eligible products, then call a REST API for current pricing, then apply local business rules. Implement all of them — do not stub any step.

3 How to Implement a Tool — Step by Step

Follow these steps for every tool file in the tools/ directory.

  1. Read agent.yaml first. Find the tool's entry. Note which MCP server, knowledge base, or API it should call. Note what actions or operations it supports. Do not guess — the yaml is the source of truth.
  2. Define the input parameters. What information does the tool need from the LLM to do its job? Create a typed dataclass with one field per parameter. Every field should have a type annotation and a sensible default. Do not use a bare pass or empty dataclass.
  3. Write the INPUT_SCHEMA. This is the OpenAI function parameters schema. Every parameter from the dataclass must appear here with a description. Use enum for fields with a fixed set of values. This is what the LLM reads to understand what arguments to pass.
  4. Write the TOOL_SPEC. Wrap INPUT_SCHEMA in the full OpenAI tool format: {"type": "function", "function": {"name": ..., "description": ..., "parameters": INPUT_SCHEMA}}. The description must explain what the tool does and when the LLM should call it.
  5. Implement the backend call in _execute(). This is the only place where the real work happens. Match the resource type to the correct calling pattern (see Section 4). If the tool supports multiple actions, use if/elif branching on the action field — one branch per backend operation.
  6. Add the execute(**kwargs) entrypoint. The agent calls tools by passing keyword arguments parsed from the LLM's JSON output. This function constructs the typed args object and calls _execute().
  7. Register the tool in tools/__init__.py. Add a ToolInfo entry with the tool name, its TOOL_SPEC, and its execute function. It will not be available to the agent until this is done.
  8. Write tests. Mock the backend layer (HTTP client, vector search client, database client). Assert that the correct backend operation is called with the correct arguments. Assert that errors propagate correctly. Do not assert stub values.

4 Backend Calling Patterns by Resource Type

MCP Server

Create a shared HTTP client module (e.g. tools/mcp_client.py) that handles the JSON-RPC 2.0 transport. Store all server URLs as constants in that module. Tool files import the client function and their server URL — they never import requests directly.

MCP protocol specifics: method is always "tools/call", arguments go under params.arguments, response comes back in a result.content[] array of typed text/json blocks. The shared client must unwrap this automatically.

Strip None values before sending — MCP servers often reject unknown optional fields.

REST API

Similarly, create a shared HTTP client or base class for the API. Store base URLs and auth header logic in one place. Tool files call helper functions, not raw requests.get(). Handle authentication (API key, OAuth token, bearer token) in the shared layer, not per-tool.

Always set a timeout. Always call raise_for_status() before reading the response. Return a plain dict — not the raw response object.

Knowledge Base / Vector Search

Use the Databricks VectorSearchClient or VectorSearchRetrieverTool from databricks_openai. The index name comes from agent.yaml under knowledge_bases. The embedding model is also declared there — use it.

A knowledge base tool typically takes a natural language query as input and returns a list of relevant document chunks with their sources. The LLM uses these chunks to ground its response. Always include the source/document reference in the return value so the agent can cite it.

Unity Catalog UDF

Use UCFunctionToolkit from databricks_openai and get_uc_function_client() from unitycatalog.ai.core.base. The UDF name follows the pattern catalog.schema.function_name. Arguments are passed as a dict to execute_function(). Check function_result.error before reading function_result.value.

Database / Delta Table

Use the Databricks SQL connector or Spark SQL via DatabricksSession. Parameterize all queries — never format user input directly into a SQL string. Return results as a list of dicts or a summary dict — not a DataFrame object, which cannot be serialized to JSON for the LLM.

Local / Computed

If no external call is needed, implement the logic directly in _execute(). Still define a proper dataclass, INPUT_SCHEMA, and TOOL_SPEC — the LLM needs to know the parameters even if the implementation is local.

5 The Agent Framework — Rules for agent.py

Class structure

The agent class must extend mlflow.pyfunc.ResponsesAgent and implement:

  • predict(request) — collects all streamed events and returns a ResponsesAgentResponse
  • predict_stream(request) — yields ResponsesAgentStreamEvent objects as they are produced

LLM client initialization

Initialize the OpenAI-compatible client lazily inside a method, not in __init__. WorkspaceClient holds threading locks that cloudpickle cannot serialize. If the client is created at instantiation time, MLflow will fail to serialize the model.

Tool execution

The agent receives tool calls from the LLM as JSON. Parse the arguments with json.loads() inside a try/except — the LLM occasionally produces malformed JSON. Catch all exceptions from tool execution and return an error string rather than crashing the agent loop.

Agent loop pattern

The loop should be event-driven: check the type/role of the last message to decide what to do next. If the last message is an assistant response with no tool calls, stop. If it is a function_call, execute the tool. Otherwise, call the LLM. Cap iterations with a max_iter guard and emit a clear message if reached.

System prompt

Insert the system prompt as the first message in every predict_stream() call. Do not rely on the caller to include it. The system prompt must describe the agent's workflow, its tools, and any compliance or business rules it must follow.

MLflow tracing

Decorate tool execution methods with @mlflow.trace(span_type=SpanType.TOOL). Call mlflow.openai.autolog() at module level. This enables the Databricks AI observability features and is required for production debugging.

6 Deploying to Databricks — Rules and Known Pitfalls

Before you deploy

  • All tests must pass — run make test
  • DATABRICKS_HOST and DATABRICKS_TOKEN must be set in your environment
  • The wheel must be built — run make build (or make deploy-dev which builds first)

Three rules for log_model()

RuleWhy
Do not pass a signature= parameter ResponsesAgent sets its signature automatically. Passing one raises MlflowException.
input_example must use the key "input", not "messages" ResponsesAgentRequest uses the Responses API schema (input), not the Chat Completions schema (messages). Using the wrong key fails Pydantic validation.
Always pass resources=[DatabricksServingEndpoint(endpoint_name=...)] Without this, the serving container cannot authenticate to the LLM endpoint at serve time.

Serving endpoint naming

Databricks does not allow updating an existing non-agent serving endpoint to serve a ResponsesAgent model. If the endpoint was previously created for a plain model, you must use a new name. The convention is {project_name}_agent.

Similarly, if you change the agent framework (e.g. LangChain → ResponsesAgent), always create a new endpoint. You cannot update the existing one in place.

Dependency management

Runtime dependencies go in pyproject.toml under [project] dependencies. Do not pin exact versions — use loose constraints (>=, not ==). Packages already pre-installed on Databricks serverless (mlflow, pydantic, pyarrow, databricks-sdk) should go in [project.optional-dependencies] dev only, not in runtime dependencies.

7 Testing Requirements

What every tool test suite must cover

TestWhat to assertMocking needed?
Tool is callable callable(tool_function) is True No
Schema is valid INPUT_SCHEMA is a dict with a properties key No
Default args are correct Dataclass defaults match the expected behavior No
Each action routes to the correct backend operation The right function/method/endpoint was called with the right arguments Yes — mock the backend client
The execute(**kwargs) entrypoint works Keyword arguments are correctly passed through to _execute() Yes — mock the backend client
Errors propagate Exceptions from the backend are not swallowed silently Yes — mock raises an exception

What to mock

Mock at the backend client boundary — not at the tool function level. For MCP tools, mock call_mcp_tool. For REST tools, mock the HTTP client. For vector search tools, mock the search client. Never mock _execute() itself — that is the function being tested.

Banned patterns
  • Do not assert result.get("_stub") is True — that validates broken behavior
  • Do not skip the error propagation test — silent failures are worse than crashes
  • Do not use @patch("agent.tool_map") — test tools in isolation, not through the agent loop

9 What to Encode in Atlas for Future Agents

Future agents generated by Atlas should be concrete and working from the start — not a set of stubs requiring a second implementation pass. The following must be encoded into Atlas as reusable Blueprints.

Agent Framework Blueprint

A reusable template for agent.py that extends ResponsesAgent, uses lazy client initialization, implements streaming, and follows the event-driven loop. This blueprint should replace the current Atlas-generated PythonModel template. Include the known log_model() constraints as comments in the generated code.

System Prompt Blueprint

Each domain (wealth management, insurance, lending, HR, etc.) should have a System Prompt Blueprint that encodes the agent's pipeline stages, compliance rules, tool descriptions, and decision gates. When an agent is created for a domain, the matching blueprint is injected — not a generic placeholder.

Tool Adapter Blueprints — one per resource type

Atlas should have a separate blueprint for each resource type (MCP, REST API, knowledge base, UC UDF, database). When a tool is added to an agent, Atlas selects the correct blueprint based on the declared resource type and generates:

  • A typed dataclass with fields derived from the tool's declared parameters
  • A populated INPUT_SCHEMA and TOOL_SPEC
  • A real _execute() with the correct calling pattern for that resource type
  • An execute(**kwargs) entrypoint
  • A test file that mocks the backend client and covers routing, args, and errors

The result: zero stubs. An agent generated by Atlas should be deployable and functional immediately after generation, with tests that pass.

Success criterion An Atlas-generated agent is complete when: all tests pass, make deploy-dev succeeds on the first try, and a real end-to-end query returns a meaningful response from the actual backends — not fabricated data.
