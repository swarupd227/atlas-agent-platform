IAM-AGT-701: Termination Intake Agent
Section	Specification
Description	Monitors SailPoint lifecycle events for termination triggers. When a SailPoint identity transitions to “Terminated” lifecycle state, this agent activates and creates a Partner Portal Removal Case. Validates against HR system (Workday) to confirm termination is legitimate and not a data error.
Trigger	SailPoint lifecycle event: identity.lifecycleState = “Terminated” AND identity.department IN [“Global Trade Settlements”, “APAC Settlements”, “Fixed Income Ops”, “Equity Settlements”, “Derivatives Ops”, “Settlement Technology”]
Workflow Steps	1. Receive termination event from SailPoint 2. Call SailPoint get_identity_cube to retrieve full identity profile 3. Validate termination date, type, and manager confirmation 4. Determine priority: Involuntary = CRITICAL (remove within 2 hours), Voluntary = HIGH (remove within 24 hours) 5. Create Removal Case with case ID, employee details, priority, SLA deadline 6. Pass case to IAM-AGT-702 Portal Discovery Agent
Ontology	Input: SailPoint termination event (identity_id, lifecycle_state, termination_date, termination_type) Output: RemovalCase (case_id, employee_id, employee_name, department, priority, sla_deadline, portals: [])
Eval Dataset	50 test cases: 25 voluntary resignations, 15 involuntary terminations, 5 retirements, 5 edge cases (same-day termination, contractor conversion, inter-company transfer). Expected: 100% correct priority classification, 0% false positives on transfers.
Regulatory	SEC Rule 17a-4: All termination processing records retained 6 years WORM. FINRA 3110: Supervisory procedures must document automated termination processing. FCA SM&CR: Senior Manager notification if terminated employee had FCA-regulated function.
IAM-AGT-704: Access Removal Executor Agent (Primary Agent)
Section	Specification
Description	The core execution agent. Removes employee access from partner portals using portal-specific adapters. Each partner portal has a different architecture, so the agent selects the correct removal method per portal. Handles 4 removal patterns: (1) API-based automatic removal, (2) Certificate revocation, (3) Token invalidation, (4) Manual removal via ServiceNow ticket.
Portal Adapters	DTCC Portal: SAML SSO → call DTCC Admin API to deactivate user Euroclear EasyWay: Client Certificate → revoke certificate via Euroclear PKI API Clearstream CASCADE: SWIFT Token → invalidate token via SWIFT Alliance API Bloomberg TOMS: Bloomberg Auth → call Bloomberg TOMS Admin API SWIFT Alliance Lite2: PKI Certificate → revoke PKI cert + deactivate operator profile ICE Link: Username/Password → call ICE Link User Management API to disable HKEX CCASS: RSA SecurID → call HKEX Admin API (if available) or manual ticket Markitserv: Username/Password → NO API support → always manual ticket
Workflow Steps	1. Receive portal list from AGT-702 with trade hold flags from AGT-703 2. For each portal, determine removal method from adapter registry 3. Check risk level: LOW/MEDIUM = auto-execute; HIGH = execute with logging; CRITICAL = require Ops Manager approval before execution 4. Execute removal via appropriate adapter 5. If portal unreachable: retry 3x with exponential backoff (5min, 15min, 60min), then create ServiceNow ticket for manual follow-up 6. If portal has no API: create ServiceNow ticket with step-by-step manual removal instructions including screenshots 7. If portal has pending trades (hold flag): schedule removal for T+1 after last settlement date 8. Record execution result per portal (success/failure/deferred/manual/pending_approval) 9. Pass results to AGT-705 for verification
Exception Handling	Portal Unreachable: Retry 3x → ServiceNow ticket (SNW category: “Partner Portal Access Removal – Retry”) → Splunk monitoring rule to alert when portal recovers Pending Trades: Hold removal → schedule post-settlement → daily check until trades settle → auto-execute removal Admin/Critical Access: Route to Ops Director for approval → if 2-person approval required (SWIFT Security Officer), route to CISO additionally → execute only after both approve Manual Portal: Generate ServiceNow ticket with: portal URL, login instructions, navigation path to user management, employee identifier in portal, required actions, verification steps
Autonomy Model	Adaptive Autonomy: • LOW risk portals (ICE Link, Clearstream viewer): Full Auto — remove immediately • MEDIUM risk portals (DTCC, Euroclear operator): Full Auto with enhanced logging • HIGH risk portals (SWIFT operator, DTCC admin): Execute with Ops Manager notification • CRITICAL risk portals (Bloomberg TOMS admin, SWIFT Security Officer): BLOCK until explicit approval from Ops Director (+ CISO for Security Officer roles)
Eval Dataset	100 test cases across all portal types: 40 clean removals, 15 portal unreachable, 15 pending trades, 10 admin access requiring approval, 10 manual-only portals, 10 mixed scenarios (multiple exceptions on same employee). Expected: 100% correct adapter selection, 100% correct risk classification, 0 unauthorized removals of critical access without approval.
Regulatory	SEC Rule 17a-4: Removal timestamp per portal recorded in WORM. FINRA 3110: Automated removal constitutes a supervisory control — must be documented in WSPs. SOX 404: Removal verification is a testable internal control. FCA: SM&CR-regulated role removals require Senior Manager notification.
