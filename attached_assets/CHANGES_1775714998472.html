4. Recommendations: Improving the Replit Prompt for Production-Ready Output

The scaffold was directionally correct but needed significant repair work before it could deploy. The following prompt additions would close the gap:

4.1 Explicitly require LangChain tool contract

"Each tool must be decorated with @tool from langchain_core.tools so it can be used with llm.bind_tools(). Separately export a plain _execute(args) function and an Args dataclass with default values so unit tests can patch _execute without importing LangChain."

4.2 Require stub returns, not NotImplementedError

"All unimplemented tool branches must return a structured dict with "_stub": True and a "message" field explaining what to integrate. Never raise NotImplementedError — it breaks the agent loop and all tests."

4.3 Specify Databricks deployment type explicitly

"Use spark_python_task in databricks.yml, not python_wheel_task. Use ${bundle.target} (not ${bundle.environment}). Set spark_version to 15.4.x-cpu-ml-scala2.12 (ML runtime). Include a libraries section with all PyPI packages needed on the cluster."

4.4 Require argparse for all CLI entry points

"Any script referenced in an MLproject entry point with parameters must parse those parameters via argparse. The MLproject command format python script.py --param {value} requires the script to declare the flag."

4.5 Require correct MLflow pyfunc predict signature

"predict(self, context, model_input, params=None) must handle both pandas.DataFrame (served endpoint) and dict (mlflow.evaluate) inputs. Response must filter to isinstance(m, (AIMessage, ToolMessage)) only — not hasattr(m, 'content')."

4.6 Require explicit pip_requirements list

"In mlflow.pyfunc.log_model(), pass an explicit list of pinned package strings to pip_requirements. Never use ['-r requirements.txt'] — the relative path is not available when the model artifact is loaded from Unity Catalog."

4.7 Require correct .env.example for Databricks projects

"For Databricks projects, .env.example must include DATABRICKS_HOST, DATABRICKS_TOKEN, MLFLOW_TRACKING_URI=databricks, and any Unity Catalog references. Do not include ANTHROPIC_API_KEY — model access goes through the Databricks endpoint, not direct Anthropic API."

4.8 Require Unity Catalog placeholder guidance

"In config.yaml, any Unity Catalog 3-part name (catalog.schema.model) must use angle-bracket placeholders like <catalog>.<schema>.model_name with a comment explaining what to fill in. The literal strings 'catalog' and 'schema' are not valid Unity Catalog names."

4.9 Require tests/__init__.py

"Always create tests/__init__.py (empty) so that pytest can resolve package-level imports consistently across environments."

4.10 Lock all dependency versions consistently across all files

"requirements.txt, conda.yaml, databricks.yml libraries section, and agent.py pip_requirements list must all specify identical pinned versions. Include pydantic>=2.0,<3 and pyarrow>=14 whenever langchain-core>=0.3 or mlflow>=2.12 are used."

6. Additional Replit Prompt Instructions (Deployment-Specific)

These cover gaps discovered only during actual databricks bundle deploy and endpoint creation — not visible from static code review.

6.1 Never use variable interpolation in DAB authentication fields

"In databricks.yml, never use ${DATABRICKS_HOST} in workspace.host or any authentication field. DAB reads DATABRICKS_HOST and DATABRICKS_TOKEN from environment variables automatically. Variable interpolation is only supported in non-auth fields such as spark_env_vars, root_path, and job names."

6.2 Always use ~/ prefix for root_path in development targets

"Set root_path: ~/.bundle/<bundle-name>/${bundle.target}. DAB enforces that development-mode deployments are user-scoped. Paths under /Shared/ are rejected unless they contain the current username."

6.3 Default to serverless compute — detect workspace type before generating cluster config

"Before generating a job_clusters block with new_cluster, check whether the target workspace supports classic clusters. Many newer Databricks workspaces are serverless-only. For serverless, generate an environments block with environment_key and spec.dependencies instead of a job_clusters block. Do not include pre-installed packages (mlflow, databricks-sdk, pyarrow, pydantic) in serverless dependencies."

6.4 Use code-based MLflow logging for any model that uses LangChain or network clients

"Never pass a Python object instance to mlflow.pyfunc.log_model(python_model=instance) if the model class or its module-level globals reference ChatDatabricks, OpenAI, or any network client. These hold threading locks that cloudpickle cannot serialize. Always use code-based logging: python_model=__file__ with mlflow.models.set_model() registered at module level."

6.5 Fix __file__ for Databricks serverless exec context

"Databricks serverless spark_python_task runs scripts via exec(compile(src, filename, 'exec')) inside an IPython kernel. __file__ is not set in this context. Add this guard at the top of any script that uses __file__:
if '__file__' not in globals(): __file__ = sys._getframe(0).f_code.co_filename"

6.6 Always use __file__-relative paths for config files

"Never use open('config.yaml') with a bare relative path in MLflow model code. The serving container's working directory is unpredictable. Always use os.path.join(os.path.dirname(os.path.abspath(__file__)), 'config.yaml'). Also include the config file in code_paths when calling log_model so it is bundled with the model artifact."

6.7 Always define an explicit MLflow model signature for Unity Catalog

"Unity Catalog rejects model registration without both input and output signatures. Never rely on signature auto-inference — it calls the model at registration time and will fail if the LLM endpoint does not exist. Always define the signature explicitly using ModelSignature with Schema and ColSpec. For chat agents use Array(Object([Property('role', DataType.string), Property('content', DataType.string)])) for both inputs and outputs."

6.8 Verify Unity Catalog catalog and schema names before hardcoding

"Never use placeholder names like main, catalog, or default without verifying they exist. Instruct the user to run SHOW CATALOGS and SHOW SCHEMAS IN <catalog> in Databricks SQL before generating registered_model_name. Different workspaces have different default catalog names — main is common but not universal."
